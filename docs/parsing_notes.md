# Parsing Notes — PDF Irregularities Found

> **Purpose**: Documents every irregularity found during PDF inspection.
> This file is gold for whoever writes the approach document or extends the parser.

---

## Inspection Method

- **Tool**: PyMuPDF 1.28.0 (`fitz`) with `page.get_text("dict")` for per-span metadata
- **Test PDF**: `data/test_manual.pdf` (generated by `tests/generate_test_pdf.py`)
- **Modeled after**: Medical device user manuals (blood glucose monitors) which commonly
  exhibit all the irregularities documented below

## Font Structure Observed

| Role              | Font Size | Bold  | Font Name          |
|-------------------|-----------|-------|--------------------|
| Document title    | 18.0      | Yes   | NimbusSans-Bold    |
| H1 (chapters)     | 15.0      | Yes   | NimbusSans-Bold    |
| H2 (sections)     | 13.0      | Yes   | NimbusSans-Bold    |
| H3 (subsections)  | 11.5      | Yes   | NimbusSans-Bold    |
| H4 (deep)         | 10.5      | Yes   | NimbusSans-Bold    |
| Body text         | 10.0      | No    | NimbusSans-Regular |
| Table cells       | 9.0       | Mixed | NimbusSans-*       |

**Key observation**: Heading detection relies on **both** font size **and** bold flag.
Body text at 10.0 is never bold; headings at 10.5+ are always bold. This gives us a
clean separation without overlap.

## Irregularity #1: Out-of-Order Subsections

**Location**: Page 2, Section 3 (Operation)

**What happens**: Section `3.4 Auto Shutoff` appears at y=257.4 on page 2,
while `3.3 Result Display` appears later at y=323.6 on the same page.

**Impact**: A naive parser that sorts children by section number would
reorder them, losing the author's intended reading flow.

**Parser behavior**:
- Children are stored in **reading order**, not numeric order
- An `OUT_OF_ORDER` irregularity is logged
- `3.3.1 Display Symbols` is correctly nested under `3.3`, not `3.4`

**Why this happens in real manuals**: Late insertions, editorial reorganization,
or regulatory amendments where renumbering the entire document would trigger
a re-review cycle.

## Irregularity #2: Skipped Heading Levels

**Location**: Page 1, under Section 2.1 (Analyzer Unit)

**What happens**: Section `2.1.1.1 Electrode Specifications` (level 4) exists
without any `2.1.1` (level 3) parent. The heading jumps from level 2 to level 4.

**Impact**: A parser that requires strict parent-child level adjacency would
either crash or misplace this node.

**Parser behavior**:
- Node is placed under nearest existing ancestor (`2.1` at level 2)
- A `SKIPPED_LEVEL` irregularity is logged noting the missing `2.1.1`
- The node retains its true level (4) for structural accuracy

**Why this happens in real manuals**: Sections copied from other documents,
template-driven content where intermediate levels were removed, or
numbering inherited from a standard (e.g., ISO 15197 references).

## Irregularity #3: Duplicate Section Numbering

**Location**: Page 3, Section 4 (Troubleshooting)

**What happens**: Two sections are both numbered `4.2`:
1. `4.2 Error Code Reference` (at y=175.2) — contains an error code table
2. `4.2 Warning Indicators` (at y=326.4) — contains body text

**Impact**: A parser using section numbers as unique keys would silently
overwrite the first occurrence.

**Parser behavior**:
- First occurrence keeps `4.2` as its section number
- Second occurrence is renamed to `4.2_dup1`
- A `DUPLICATE_NUMBER` irregularity is logged with both titles
- Both nodes have different content hashes (critical for versioning)

**Why this happens in real manuals**: Copy-paste errors during authoring,
table of contents not synced with body, or separate teams authoring
different sections without coordination.

## Irregularity #4: Tables with Mixed Font Styles

**Location**: Page 3 (Error Codes) and Page 4 (Maintenance Schedule)

**What happens**: Tables use 9.0pt font for all cells, but header rows are
bold (`NimbusSans-Bold`) while data rows are regular (`NimbusSans-Regular`).
Multiple spans appear at the same y-coordinate (one per column).

**Detection heuristic**:
- Font size ≤ 9.5 (TABLE_MAX_SIZE)
- Multiple spans at the same y-coordinate (within 2.0pt tolerance)
- Bold spans at the same y as non-bold → still merged for table detection
- First bold row → header; subsequent non-bold rows → data

**Tables found**:

|------|-----------------------|---------------------------|
| E001 | Strip not detected    | Reinsert test strip       |
| E002 | Insufficient sample   | Apply more blood          |
| E003 | Temperature out of range | Move to 15-40C environment |
| E004 | Battery critically low | Replace batteries         |
| E005 | Sensor malfunction    | Contact service           |

### Maintenance Schedule (under 5.1)
| Task               | Frequency  | Procedure                        |
|--------------------|------------|----------------------------------|

| Run QC test        | Daily      | Use Level 1 and Level 2 controls |

## Irregularity #5: Multi-Page Content

**Location**: Content spans pages 1–4

**What happens**: Section boundaries don't align with page boundaries.
Section 2 starts on page 1 and its subsection `2.1.1.1` finishes at the
bottom of page 1.

**Parser behavior**: Page number is tracked per-node based on where the
heading first appears (page_number field). Content may span multiple pages
but the node records only its starting page.

## Parser Architecture Summary

```
PDF File
  │
  ▼
[1] Extract raw spans (PyMuPDF dict mode)
    → RawSpan objects with font metadata
  │
  ▼
[2] Merge spans into logical lines
    → Same-y + same-font spans concatenated
    → Small-font same-y spans merged for tables
  │
  ▼
[3] Detect and extract tables
    → Clustered by y-coordinate
    → Header/data separation by bold flag
  │
  ▼
[4] Classify lines (heading level or body text)
    → Section number regex + font size thresholds
  │
  ▼
[5] Build tree with stack-based insertion
    → Out-of-order: preserved in reading order
    → Skipped levels: insert under nearest ancestor
    → Duplicates: disambiguated with _dupN suffix
  │
  ▼
[6] Compute content hashes (SHA-256 per node)
    → hash(section_number | title | content)

Result: ParsedDocument with tree + irregularity log
```

## Why PyMuPDF over alternatives

| Library     | Pros                                          | Cons                                    |
|-------------|-----------------------------------------------|-----------------------------------------|
| **PyMuPDF** | Per-span font metadata, 10-50× faster, bbox   | Table extraction is manual              |
| pdfplumber  | Better table extraction, good coordinates      | Slower, less font detail                |
| pdf2image+OCR | Works on scanned docs                       | Overkill for digital PDFs, slow         |
| LLM-based   | Handles any layout                            | Non-deterministic, expensive, slow      |

**Decision**: PyMuPDF with custom table detection heuristics. The font metadata
is critical for heading classification, and our table detection (y-coordinate
clustering + bold detection) handles the simple grid tables found in medical manuals.

For complex tables (merged cells, nested headers), a future enhancement could
layer pdfplumber's table extraction on top of PyMuPDF's span extraction.
